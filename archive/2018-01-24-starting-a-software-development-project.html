--- title: Starting a Software Development Project date: 2018-01-24 20:00:00 +0100 ---

Context

A bit more then a year after the Software Apprenticeship started, we covered a lot of topics.
But still a major topic was missing: How to use that (theoretical) knowledge for building a working Software Solution.
Luckily we caught a break: A friend challenged us to build an App to help her in doing some boring recurring administrative work. This was an nice opportunity to explain and experience how a Software Project gets started.

Important questions first

When a project gets started there are usually a lot of unknowns and questions, which needs answers. All or most of them needs to be addressed, but not all of them have the same priority and a very interesting aspect about all these questions is: How do we keep track of the questions and answers.

How are we going to call this project?
Once something has a name it becomes a identity and we can speak more easily about it, so naming this is important, but naming things is hard and it's best done as a process.
Eventually things like marketing, branding, availability of a domain name and availability on Social Media become important, but not just yet.
So start with some 'internal project name' and don't worry about the rest yet.
Suggestions for codenames:
  • Combine some word of the address of the client from what three words
  • Some random generated name like docker containers
In which language are we going to write the documentation / Software Architecture Document
The short answer: Use English!
The longer answer: For our project we want to have a 'Ubiquitous Language' All stakeholders and participants in the project should be able to communicate about project and should not be hindered by their capacity to speak / write in the language used in the Ubiquitous Language. So under some conditions perhaps you should switch to the native language of you product owner.
Where are we going to store the documentation?
Documentation should be under version control as well!
So logical places for documentation are:
  • A markdown in GIT
  • Some wiki with History / Version Control
  • Word with track changes

What are we going to document?

Fortunately there are some good model available to identify what needs to be investigated and documented related to a Software Product.
I prefer to write a Software Architecture Document based upon '4+1 Architectural View Model'
This structure help me to answer the following questions:

What are we going to build?
In order to get an general impression what the system is going to do, I find it useful to draw a 'System Landscape Diagram', which show and identifies the actors, the other software systems involved and their relationship to the 'Software System'.
A 'System Landscape Diagram' is one of the Core diagrams of the C4 Model

The Scenario's' should be written as feature files using Gherkin, so that they can be reused for automatic tests using Cucumber
If a more detailed / formal documentation is needed for features, you could look at 'Use Case Descriptions' although they are a bit out of fashion.
How are we going to build this project?
The Development view should give answer to the following questions: Where are going to store the project code?
In which language are we going to develop this project?
Which frameworks are we going to use?
How do we test, build, deploy the project?
How do we track our bugs, features, improvements?

This could be written down as a seperate chapter of the SAD or in the README.md of the project.
What is the structure of the appliation?
This is a combination of the Logical View with information like a Entity-Relationship-Diagram or database schema and Process View with Sequence Diagrams, State Diagrams and Activity Diagrams
How does the application get deployed in various environments?
A Physical View with a deployment diagram.
A 'Container Diagram' from the C4 Model works pretty good for this!

Ubiquitous Language

As mentioned before: Each project should use a 'Ubiquitous Language'
In order to make adoption of the Ubiquitous Language for the project as easy as possible, the project team should create a Glossary, which identifies all key terms or concepts and adds a description to the clarify if needed.
